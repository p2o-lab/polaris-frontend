About Polaris

Polaris is a process orchestration layer (POL) tool for orchestrating process equipment assemblies (PEA) in a modular plant proudly presented by the P2O-Lab from TU Dresden.

It provides mechanisms to observe and control services and process values provided by PEAs and perform recipes defined in a JSON. You can easily integrate PEAs into Polaris by loading its interface descriptions stored in a JSON file. This JSON file can automatically be generated from a MTP of this module.

Modules

Polaris can control multiple PEAs, also known as modules. Modules are loaded via JSON files which describe the necessary communication variables to interact with.

The Module JSON contains all information necessary to interact with the module. This information is automatically extracted from an MTP (module type package) of the module.

Service

Each service consists of a finite state machine which is used to control the service. For that, commands on the modules view are used to trigger state transitions.

You can send commands (e.g. start, pause, stop etc.) to specific services by clicking the appropriate buttons. If the buttons are disabled (greyed out), the module is not ready to perform this command. All services follow the state machine defined in the VDI/VDE/NAMUR 2658. So only some commands are available in specific states. Furthermore, the module can also have more restrictions to start or restart a service: 1) Additional constraints for the service parameters (change the parameters, then the appropriate command should be available) 2) interlocks between services (bring the service that interlocks the desired one to IDLE; then this service should be able to start)

Recipes

Recipes are also loaded via JSON. They describe in a declarative way which steps should be executed one after another and which actions should be taken in each step (e.g. starting or stopping a service). Furthermore, each step defines a set of transitions which lead under a defined condition to a next step.

Playlist

You can dynamically enqueue recipes in a playlist which then can started. This allows to define a set of recipes which will be executed one after another. The external trigger can also be used to start the playlist. If external trigger is true and the playlist is currently not running, the playlist will be started.

Settings

The settings view allows you to adapt settings of Polaris. The settings are persisted in your browser.

As first option, you can change the URI of the Polaris backend (if you want to host the frontend somewhere else than the backend). The default value of the URI refers to the same server as the frontend. The same way you can change the URI of the MTP converter which runs also as a webservice and converts MTPs into module JSON files.

Furthermore, you can enable AutoReset. If this feature is enabled, Polaris will automatically reset a service (thus, bring it to IDLE) if the service has COMPLETED. This reduces additional service calls from the user or the recipe engine. The default option of AutoReset is on.

Lastly, you can see the version of both frontend and backend. Additionally, you can shutdown the backend (which than should be automatically restarted if started via docker-compose)
